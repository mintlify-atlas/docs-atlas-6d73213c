---
title: "STM32H7 LL drivers: low-layer direct register access"
description: "STM32H7 LL drivers: inline register-level peripheral access with deterministic timing and minimal code size, usable standalone or mixed with HAL in one project."
---

The Low-Layer (LL) drivers are a set of header-only, inline C functions that map directly onto peripheral registers. Where the HAL abstracts a UART transmit into a state-machine with timeout and DMA orchestration, the LL equivalent is a single `LL_USART_TransmitData8()` call that writes one byte to the transmit data register and returns immediately. The trade-off is explicit: less portable code in exchange for deterministic timing and smaller footprint.

## LL vs HAL — when to use each

| Criterion | HAL | LL |
|-----------|-----|-----|
| Portability across STM32 families | High | Medium (register names are device-specific) |
| Code size | Larger (state machines, callbacks) | Minimal (inline, zero overhead) |
| Execution time predictability | Variable (polling loops, DMA setup) | Deterministic (single register write) |
| Ease of use | High (handles, callbacks, DMA chaining) | Low (you manage every step) |
| DMA support | Built-in | Manual (set up DMAMUX + DMA registers yourself) |
| Mixed use with HAL | Supported | Supported |

<Tip>
  LL drivers are ideal for time-critical ISR paths, ultra-low-power sequences where you need precise control of clock enable/disable timing, and situations where every kilobyte of flash matters. For most application-layer code, HAL is the right starting point.
</Tip>

## API naming convention

LL function names follow a strict pattern:

```
LL_<Peripheral>_<Action>[_<Qualifier>]
```

Examples:

| Function | Action |
|----------|--------|
| `LL_GPIO_SetOutputPin(GPIOB, LL_GPIO_PIN_0)` | Set pin high |
| `LL_GPIO_ResetOutputPin(GPIOB, LL_GPIO_PIN_0)` | Set pin low |
| `LL_GPIO_TogglePin(GPIOB, LL_GPIO_PIN_0)` | Toggle pin |
| `LL_USART_TransmitData8(USART3, byte)` | Write byte to TDR |
| `LL_USART_IsActiveFlag_TXE(USART3)` | Test TXE flag |
| `LL_RCC_GetSystemClocksFreq(&clocks)` | Read clock tree |

All LL functions are declared `__STATIC_INLINE` — the compiler inlines them and the linker discards unused ones, so including an LL header adds zero overhead for functions you never call.

## Standalone vs mixed use

You can use LL drivers in two ways:

<Tabs>
  <Tab title="Standalone (no HAL)">
    Define `USE_FULL_LL_DRIVER` and use LL init functions to configure peripherals from scratch. This avoids pulling in any HAL source file.

    ```c
    #define USE_FULL_LL_DRIVER
    #include "stm32h7xx_ll_gpio.h"
    #include "stm32h7xx_ll_usart.h"
    #include "stm32h7xx_ll_rcc.h"
    #include "stm32h7xx_ll_bus.h"
    ```
  </Tab>
  <Tab title="Mixed with HAL">
    Include the LL headers alongside your normal HAL includes. The HAL and LL share the same device header (`stm32h7xx.h`) so there are no conflicts. A common pattern is to initialise a peripheral with HAL and then use LL calls inside the ISR for minimum latency.

    ```c
    #include "stm32h7xx_hal.h"
    #include "stm32h7xx_ll_usart.h"   /* LL header on top of HAL */
    ```
  </Tab>
</Tabs>

## GPIO example

```c
#define USE_FULL_LL_DRIVER
#include "stm32h7xx_ll_gpio.h"
#include "stm32h7xx_ll_bus.h"

void LL_GPIO_Example(void)
{
    /* Enable GPIOB peripheral clock */
    LL_AHB4_GRP1_EnableClock(LL_AHB4_GRP1_PERIPH_GPIOB);

    /* Configure PB0 as push-pull output, no pull, low speed */
    LL_GPIO_InitTypeDef gpio_init = {0};
    gpio_init.Pin        = LL_GPIO_PIN_0;
    gpio_init.Mode       = LL_GPIO_MODE_OUTPUT;
    gpio_init.OutputType = LL_GPIO_OUTPUT_PUSHPULL;
    gpio_init.Pull       = LL_GPIO_PULL_NO;
    gpio_init.Speed      = LL_GPIO_SPEED_FREQ_LOW;
    LL_GPIO_Init(GPIOB, &gpio_init);

    /* Toggle in a loop */
    while (1)
    {
        LL_GPIO_TogglePin(GPIOB, LL_GPIO_PIN_0);
        /* Delay here … */
    }
}
```

The `LL_GPIO_InitTypeDef` structure mirrors the HAL `GPIO_InitTypeDef` but uses `LL_GPIO_*` constants. When `USE_FULL_LL_DRIVER` is not defined, `LL_GPIO_Init()` is not compiled in — only the single-function inlines remain.

## USART example

```c
#define USE_FULL_LL_DRIVER
#include "stm32h7xx_ll_usart.h"
#include "stm32h7xx_ll_rcc.h"
#include "stm32h7xx_ll_bus.h"
#include "stm32h7xx_ll_gpio.h"

void LL_USART_Example(void)
{
    /* 1. Enable clocks */
    LL_APB1_GRP1_EnableClock(LL_APB1_GRP1_PERIPH_USART3);
    LL_AHB4_GRP1_EnableClock(LL_AHB4_GRP1_PERIPH_GPIOD);

    /* 2. Configure PD8 (TX) and PD9 (RX) as AF7 */
    LL_GPIO_InitTypeDef gpio = {0};
    gpio.Pin        = LL_GPIO_PIN_8 | LL_GPIO_PIN_9;
    gpio.Mode       = LL_GPIO_MODE_ALTERNATE;
    gpio.Alternate  = LL_GPIO_AF_7;
    gpio.OutputType = LL_GPIO_OUTPUT_PUSHPULL;
    gpio.Pull       = LL_GPIO_PULL_UP;
    gpio.Speed      = LL_GPIO_SPEED_FREQ_VERY_HIGH;
    LL_GPIO_Init(GPIOD, &gpio);

    /* 3. Configure USART3 for 115200 8N1 */
    LL_USART_InitTypeDef usart_init = {0};
    usart_init.BaudRate            = 115200U;
    usart_init.DataWidth           = LL_USART_DATAWIDTH_8B;
    usart_init.StopBits            = LL_USART_STOPBITS_1;
    usart_init.Parity              = LL_USART_PARITY_NONE;
    usart_init.TransferDirection   = LL_USART_DIRECTION_TX_RX;
    usart_init.HardwareFlowControl = LL_USART_HWCONTROL_NONE;
    usart_init.OverSampling        = LL_USART_OVERSAMPLING_16;
    LL_USART_Init(USART3, &usart_init);
    LL_USART_Enable(USART3);

    /* 4. Transmit a byte — poll TXE then write TDR */
    while (!LL_USART_IsActiveFlag_TXE(USART3))
        ;
    LL_USART_TransmitData8(USART3, 'A');

    /* Wait for transmission complete before disabling */
    while (!LL_USART_IsActiveFlag_TC(USART3))
        ;
}
```

## LL init structure pattern

Every LL peripheral with complex configuration ships a corresponding `LL_<Periph>_InitTypeDef` and `LL_<Periph>_Init()` function (only available when `USE_FULL_LL_DRIVER` is defined). The pattern is:

```c
/* 1. Zero-initialise the struct so unset fields default safely */
LL_SPI_InitTypeDef spi_init = {0};

/* 2. Set only the fields that differ from the zero default */
spi_init.TransferDirection = LL_SPI_FULL_DUPLEX;
spi_init.Mode              = LL_SPI_MODE_MASTER;
spi_init.DataWidth         = LL_SPI_DATAWIDTH_8BIT;
spi_init.ClockPolarity     = LL_SPI_POLARITY_LOW;
spi_init.ClockPhase        = LL_SPI_PHASE_1EDGE;
spi_init.NSS               = LL_SPI_NSS_SOFT;
spi_init.BaudRate          = LL_SPI_BAUDRATEPRESCALER_DIV8;
spi_init.BitOrder          = LL_SPI_MSB_FIRST;

/* 3. Pass to LL_<Periph>_Init — returns SUCCESS or ERROR */
ErrorStatus err = LL_SPI_Init(SPI1, &spi_init);
```

`LL_<Periph>_Init()` returns `ErrorStatus` (`SUCCESS` = 0, `ERROR` = 1) rather than `HAL_StatusTypeDef`.

## Using LL in an ISR alongside HAL

A common mixed pattern: HAL owns the peripheral and drives the state machine; LL provides the low-latency flag checks inside the ISR:

```c
/* In the IRQ handler — use LL to avoid HAL state-machine overhead */
void USART3_IRQHandler(void)
{
    if (LL_USART_IsActiveFlag_RXNE(USART3) &&
        LL_USART_IsEnabledIT_RXNE(USART3))
    {
        uint8_t byte = LL_USART_ReceiveData8(USART3);
        ring_buffer_push(&rx_ring, byte);
    }
}
```

<Note>
  When mixing HAL and LL on the same peripheral instance, do not call `HAL_UART_Receive_IT()` and also manually enable the RXNE interrupt via LL — the HAL owns the interrupt enable bits on handles it manages. Either use the HAL callback model or bypass the HAL completely with LL.
</Note>
