The `Projects/` directory is the primary entry point for working with STM32CubeH7 firmware. Each subdirectory maps to a specific evaluation board or Nucleo kit, and within it you will find ready-to-build projects organized by type — examples, applications, demonstrations, and templates — each pre-configured for three major IDEs.

## Directory layout

Every board directory follows the same structure:

```text
Projects/
└── <BoardName>/
    ├── Examples/            # HAL-based peripheral examples
    ├── Examples_LL/         # Low-layer (LL) driver examples
    ├── Examples_MIX/        # Examples mixing HAL and LL APIs
    ├── Applications/        # Middleware integration projects
    ├── Demonstrations/      # Full-featured demo applications
    └── Templates/           # Bare skeleton for new projects
        Templates_LL/        # LL-based bare skeleton
```

<Note>
Not every board provides all six subdirectory types. For example, `STM32H747I-DISCO` ships `Applications/`, `Examples/`, and `Templates_LL/` but no HAL-only `Templates/` or `Examples_LL/`. Check the individual board directory before expecting a specific folder.
</Note>

## Supported IDEs

Every project directory includes toolchain-specific subdirectories with fully configured project files. No manual project setup is required — open the relevant file and build.

<Tabs>
  <Tab title="STM32CubeIDE">
    STM32CubeIDE is the recommended Eclipse-based IDE from STMicroelectronics. Project files are stored in `STM32CubeIDE/`.

    ```text
    Templates/STM32CubeIDE/
    ├── .project
    ├── .cproject
    ├── Example/
    ├── STM32H743ZITx_FLASH_DTCMRAM.ld
    └── STM32H743ZITx_FLASH_RAM_D1.ld
    ```

    Open the project via **File → Import → Existing Projects into Workspace** and point at the `STM32CubeIDE/` folder. Two linker scripts are provided: one places the default RAM region in DTCM RAM (`@0x20000000`) and the other in D1 AXI-SRAM (`@0x24000000`).
  </Tab>
  <Tab title="Keil MDK-ARM">
    Keil MDK-ARM project files are stored in `MDK-ARM/`.

    ```text
    Templates/MDK-ARM/
    ├── Project.uvprojx
    ├── Project.uvoptx
    └── startup_stm32h743xx.s
    ```

    Open `Project.uvprojx` directly in Keil µVision. To switch the default RAM target between DTCM and D1-AXI SRAM, go to **Project options → Target** and toggle IRAM1 / IRAM2 accordingly.
  </Tab>
  <Tab title="IAR EWARM">
    IAR Embedded Workbench project files are stored in `EWARM/`.

    ```text
    Templates/EWARM/
    ├── Project.ewp
    ├── Project.ewd
    ├── Project.eww
    ├── startup_stm32h743xx.s
    ├── stm32h743xx_flash_dtcmram.icf
    └── stm32h743xx_flash_ram_d1.icf
    ```

    Open `Project.eww` in IAR EWARM. Two `.icf` linker configuration files correspond to DTCM RAM and D1-AXI SRAM placement. Select one via **Project options → Linker → Config**.
  </Tab>
</Tabs>

## Project categories explained

| Category | API layer | Purpose |
|---|---|---|
| `Examples` | HAL | One peripheral, one concept — easiest starting point |
| `Examples_LL` | LL (register-level) | Same concepts as HAL examples but using the low-layer API for tighter control |
| `Examples_MIX` | HAL + LL | Demonstrates mixing both APIs in a single project |
| `Applications` | HAL + Middleware | Integrates FreeRTOS, LwIP, FatFs, USB, or other middleware stacks |
| `Demonstrations` | HAL + Middleware | Polished end-to-end demos, often requiring BSP display drivers |
| `Templates` | HAL | Minimal skeleton; copy this as the starting point for a new project |
| `Templates_LL` | LL | Same minimal skeleton but using the LL API |

## Repository structure note

STM32CubeH7 uses Git submodules to track shared middleware and driver components. The `Drivers/`, `Middlewares/`, and `Utilities/` directories at the repository root are shared across all board projects. When you clone the repository, initialize submodules to ensure these components are available:

```bash
git clone https://github.com/STMicroelectronics/STM32CubeH7.git
cd STM32CubeH7
git submodule update --init --recursive
```

<Warning>
Opening a project from a clone where submodules have not been initialized will result in missing header files and build failures. Always run `git submodule update --init` after cloning.
</Warning>
